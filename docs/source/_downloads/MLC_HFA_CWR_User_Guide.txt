## THE MLC‚Äôs CWR User Guide

Common Works Registration (CWR) is a‚ÄØglobally‚ÄØrecognized‚ÄØstandard for the registration and revision of musical works. CWR enables the efficient tracking of registration status and allows for better communication between music publishers and Mechanical Rights Societies and Performing Rights Organizations (PROs). CWR is one way you can submit your mechanical claims to The MLC. CWR is typically used for submitting large volumes of work registrations or updates (hundreds or thousands) at once and is a common tool for organizations managing large catalogs. CWR specifications and user manuals can be found on the CISAC website https://members.cisac.org/CisacPortal/documentPack20.do?method=detail&id=16 This Guide is for Members who wish to submit work registrations to The MLC using CWR formatted files. Please note that the CWR format currently only accommodates the submission of registrations that cover all mechanical usage types. If you only wish to claim certain mechanical usage types for the works you are registering (for example, if you only wish to claim digital-only mechanical usages), please advise The MLC BEFORE submitting the corresponding CWR files by contacting us here: https://portal.themlc.com/contact-us

#### ‚Ä¢ File Naming

o CISAC CWR naming convention is CWYYXXXXTST_000.V21:

ÔÇß CW = indicates CWR file

ÔÇß YY = indicates the year of generation

ÔÇß XXXX = indicates the sequence of the CWR submission. Note: the combination of sequence number and year should be unique

ÔÇß TST = CISAC-registered submitter code. This code is awarded by CISAC. The MLC can help you obtain a sender code if you do not already have one. Please contact https://portal.themlc.com/contact-us if required. This code must be included in the name of each CWR file submitted to The MLC.

ÔÇß 000 = Recipient society code

‚Ä¢ **The MLC accepts 034 (our vendor‚Äôs society code) and 000 (the code for ‚Äúmultiple societies‚Äù)

ÔÇß V21 = indicates version of CWR (We currently accept .V21 (version 2.1) or .V22 (version 2.2); once again, please contact us if you wish to use version 3.1)

o Please note: each file you submit to The MLC must have a unique CWR file name. If you submit a CWR file using a previously used CWR file name, the file will be automatically rejected.

#### ‚Ä¢ File Size

o The maximum file size we can accept is 10,000 transactions. Please see File Delivery for further details regarding how files will be ingested based on file size.

#### Technical Setup ‚Äì Works

‚Ä¢ You must provide Ownership transactions for all parties on a given work, regardless of whether they are controlled or non-controlled. Please use the following codes to identify each ownership transaction you submit:

o SPU (submitting/controlled publisher)

o SWR (submitting/controlled writer)

o OPU (other/non-controlled publisher)

o OWR (other/non-controlled writer)

‚Ä¢ The total ownership shares for all writers and publishers for each right must total either 100% plus or minus 0.06% or 0%.

‚Ä¢ Total collection shares for each work submitted must be less than or equal to 100% for mechanical shares.

‚Ä¢ Note that a tolerance of plus or minus 0.06% is allowed.

‚Ä¢ If you do not know whom the non-controlled publisher (OPU) or non-controlled writer (OWR) is, you can use the ‚Äúunknown writer‚Äù or ‚Äúunknown publisher‚Äù indicator or one of the following terms as entities (by way of example, and not limitation): UNKNOWN PUBLISHER, UNKNOWN WRITER, INCONNU EDITEUR, INCONNU COMPOSITEUR AUTEUR, COPYRIGHT CONTROL

‚Ä¢ If you use ‚Äúunknown‚Äù flags and entities for a controlled publisher (SPU) or controlled writer (SWR), such transactions will be rejected as the controlled records form the basis of your claim.

‚Ä¢ The total of collection shares cannot exceed 100% for a given right for a territory. Note that a tolerance of plus or minus 0.06% is allowed.

‚Ä¢ Submitters are welcome to include all alternate title types in their registration.

#### Technical Setup ‚Äì Interested Parties

‚Ä¢ Interested Party Identifiers

o We require unique submitter interested party identifiers across all CWR files, regardless of right type. These are identifiers assigned by the submitter to a specific writer or publisher and typically come from a submitter‚Äôs internal database.

ÔÇß Ex: If a submitter uses 000012345 as a party identifier for PUBLISHER X, then that identifier must be used across all works for submissions relating to PUBLISHER X.

‚Ä¢ IPI name numbers are strongly encouraged in the designated IPI field for submitted writers and publishers.

#### Best Practices ‚Äì NWRs (new work registrations)

‚Ä¢ It is always more efficient for The MLC to receive a work once, with the most complete information possible on the NWR (new work registration). That way, no future REVs (revisions) are needed.

‚Ä¢ ISRCs, recording artists and other recording details can be included on NWR transactions. There is not a limit on recordings (REC transaction) or performer (PER) transactions included in a CWR transaction.

#### Best Practices ‚Äì REVs (revisions)

‚Ä¢ REVs (revisions) should primarily be used for revisions to the controlled shares on a work.

‚Ä¢ We will utilize the Submitter Work # field to match to your earlier registration and process the share/title updates accordingly.

‚Ä¢ If a submitter wishes to supplement existing registrations with additional recording metadata, there is a bulk recording upload template available upon request. By using this template (rather than REV transactions), the recordings will be more readily available for matching.

‚Ä¢ There is no requirement for submitters to make updates to non-controlled splits as we rely upon the other participants on a work to claim their shares accordingly with The MLC.

‚Ä¢ Updates to writer and publisher names across existing registrations, including IPI name numbers, can be made by contacting https://portal.themlc.com/contact-us

#### Acknowledgements

‚Ä¢ Submitters will receive both 1st and 2nd Acknowledgment (ACK) files. Acknowledgment files will include The MLC Song Codes when the transaction has been accepted.

‚Ä¢ 1st ACK files

o 1st ACKs typically denote which registrations were accepted into the processing queues and which were rejected due to technical reasons. The codes a submitter would ordinarily receive could be one of the below:

ÔÇß RA (registration accepted) ‚Äì the submission was accepted into the processing system.

ÔÇß RJ (registration rejected) ‚Äì the submission was not accepted into the processing system. The submission did not meet technical requirements.

ÔÇß NP (no participation) ‚Äì no technical issue with the submission, but no acceptable territory was included on the work and thus it does not pertain to royalty collection in the USA.

ÔÇß AC (accepted with changes) ‚Äì Occasionally, works do not require review and become instantly available for matching. Those works will receive an AC code meaning the work claims have been ingested and should be visible on The MLC‚Äôs public database. The MLC song code will be included in the ACK file alongside the transaction code AC.

o 1st ACKs are typically delivered within two business days after file submission.

‚Ä¢ 2nd ACK Files

o 2nd ACK files denote registrations that have been gone through the processing queues, with a work either being created, an existing work being updated, or a work being rejected as a conflict against an existing work at The MLC.

o The ACK codes that correspond to these are:

ÔÇß AC (accepted with changes) ‚Äì the registration is complete with claims ingested and as such should be visible on The MLC‚Äôs public database. The MLC song code will be included in the ACK file alongside the transaction code AC.

ÔÇß CO (conflict) ‚Äì registration was rejected against an existing registration at The MLC. Please contact https://portal.themlc.com/contact-us if you wish to start overclaim proceedings

ÔÇß RJ (registration rejected) ‚Äì registration was rejected but, not necessarily as a conflict. These 2nd ACK RJs are uncommon and typically occur when a submitter requests that their own registration be rejected due to a mistake by them.

o 2nd ACKs are typically sent to the submitter periodically as registrations proceed through the processing queues. They are submitted in batches to correspond to the submitter‚Äôs original CWR file.

ÔÇß For example, if a submitter drops off a CWR file with 100 works, the 1st ACK files should be available within 2 days noting 100 transactions with codes RA/RJ/AC.

ÔÇß Then, 2nd ACK files will be generated with codes AC, RJs and CO periodically until all submissions have been completed.
